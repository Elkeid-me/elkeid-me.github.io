## 简介

`asm`关键字允许在 C 代码中嵌入汇编程序指令。GCC 提供两种形式的内联 `asm` 语句。基础 `asm` 语句是一个没有操作数的语句（见[基础汇编](#基础汇编——不带操作数的汇编程序指令)），而扩展 `asm` 语句（见[扩展汇编](#扩展汇编——带有-C-语言表达式操作数的汇编指令)）包括一个或多个操作数。在一个函数中混合使用 C 语言和汇编语言时，扩展形式是首选，但要在顶层[^1]包含汇编语言，就必须使用基础 `asm`。

也可以使用 `asm` 关键字来覆盖一个 C 语言符号的汇编器名称，或者将一个 C 语言变量放在一个特定的寄存器中。

## 基础汇编——不带操作数的汇编程序指令

基础 asm 语句具有以下语法：

```cpp
asm [asm修饰符] (汇编指令)
```

对于 C 语言，`asm` 关键字是一个 GNU 扩展。当使用 `-ansi` 和 `-std` 编译选项来编写没有 GNU 扩展的 C 方言时，使用 `__asm__` 而不是 `asm`（见[替代关键字](https://gcc.gnu.org/onlinedocs/gcc/Alternate-Keywords.html)）。对于 C++ 语言，`asm` 是标准关键字，但 `__asm__` 可以用于使用 `-fno-asm` 选项编译的代码。

### 修饰符

- `volatile`
  可选的 `volatile` 修饰符没有作用。所有基础 `asm` 块都隐含 `volatile`。
- `inline`
  如果使用 `inline` 修饰符，那么为了内联的目的，`asm` 语句的大小被认为是尽可能小的大小（见[asm 的大小](https://gcc.gnu.org/onlinedocs/gcc/Size-of-an-asm.html)）。

### 参数

- 汇编指令
  这是一个指定汇编器代码的字面字符串。这个字符串可以包含任何被汇编器识别的指令，包括指令。GCC 并不解析汇编器指令本身，也不知道它们是什么意思，甚至不知道它们是否是有效的汇编器输入。

  可以把多条汇编指令放在一起，放在一个 `asm` 字符串中，用系统的汇编代码中通常使用的字符分开。一个在大多数地方都有效的组合是用换行符来断行，加上一个制表符（写成 `\n\t`）。一些汇编程序允许用分号作为行的分隔符。然而，请注意，有些汇编器方言使用分号来开始注释。

### 备注

使用扩展 `asm`（见[扩展汇编](#扩展汇编——带有-C-语言表达式操作数的汇编指令)）通常会产生更小、更安全、更有效的代码，在大多数情况下，它是比基础 `asm` 更好的解决方案。然而，有两种情况下只能使用基础 `asm`：

- 扩展 `asm` 语句必须在 C 函数内。所以在文件范围内（“顶层”），即在 C 函数之外写内联汇编语言，必须使用基础 `asm`。可以使用这种技术来发出汇编指令，定义可以在文件其他地方调用的汇编语言宏，或者用汇编语言写整个函数。函数之外的基础 `asm` 语句不能使用任何修饰符。

- 用 `naked` 属性声明的函数也需要基础 `asm`（见[函数属性](https://gcc.gnu.org/onlinedocs/gcc/Function-Attributes.html)）。

安全地访问 C 数据和从基础 `asm` 中调用函数比它看起来要复杂得多。要访问 C 数据，最好使用扩展 `asm`。

不要期望一连串的 `asm` 语句在编译后能保持完全连续。如果某些指令需要在输出中保持连续，可以把它们放在一个单一的多指令 `asm` 语句中。请注意，GCC 的优化器可以将 `asm` 语句相对于其他代码进行移动，包括跨越跳转。

`asm` 语句可能不会执行跳转到其他 `asm` 语句中。GCC 不知道这些跳转，因此在决定如何优化时不能考虑到它们。从 `asm` 到 C 标签的跳转只在扩展 `asm` 中支持。

某些情况下，GCC 在优化时可能会重复（或删除重复的）汇编代码。如果你的汇编代码定义了符号或标签，这可能导致在编译过程中出现意想不到的重复符号错误。

**警告：** C 语言标准没有指定 `asm` 的语义，使其成为编译器之间不兼容的潜在来源。这些不兼容的情况可能不会产生编译器警告/错误。

GCC 不解析基础 `asm` 的汇编指令，这意味着没有办法向编译器传达里面发生了什么。GCC 对 `asm` 中的符号不了解，可能会把它们当作未引用的符号丢弃。它也不知道汇编程序代码的副作用，例如对内存或寄存器的修改。与某些编译器不同的是，GCC 假定不会对通
用寄存器进行修改。这个假设在未来的版本中可能会改变。

为了避免未来语义的变化和编译器之间的兼容性问题带来的复杂情况，可以考虑用扩展 `asm` 代替基础 `asm` 。有关如何进行这种转换的信息，请见[如何从基础 asm 转换到扩展 asm](https://gcc.gnu.org/wiki/ConvertBasicAsmToExtended)。

编译器将基础 `asm` 中的汇编指令逐字复制到汇编语言输出文件中，不处理方言或任何扩展 `asm` 中的 `%` 运算符。这导致了基础 `asm` 字符串和扩展 `asm` 模板之间的微小差异。例如，为了引用寄存器，你可以在基础 `asm` 中使用 `%eax`，而在扩展 `asm` 中使用 `%%eax`。

在像 x86 这样支持多种汇编方言的目标架构上，所有基础 `asm` 块都使用由 `-masm` 命令行选项指定的汇编方言（见[x86 选项](https://gcc.gnu.org/onlinedocs/gcc/x86-Options.html)）。基础 `asm` 没有为不同的方言提供不同汇编器字符串
的机制。

对于带有非空汇编字符串的基础 `asm`，GCC 假定汇编块不改变任何通用寄存器，但它
可能会读或写任何全局可访问的变量。

下面是一个 i386 的基础 `asm` 的例子：

```cpp
/* 注意，此代码在使用 -masm=intel 命令行选项时无法编译 */
#define DebugBreak() asm("int $3")
```

## 扩展汇编——带有 C 语言表达式操作数的汇编指令

使用扩展 `asm`，你可以从汇编器中读写 C 变量，并执行从汇编器代码到 C 标签的跳转。扩展 `asm` 语法使用冒号 `:` 来限定汇编器模板后的操作数参数。

```cpp
asm asm修饰符 ( 汇编模板
              : 输出操作数
            [ : 输入操作数
            [ : Clobbers ] ])
```

```cpp
asm asm修饰符 ( 汇编模板
              : 输出操作数
              : 输入操作数
              : Clobbers
              : Goto标签 )
```

其中，在最后一种形式中，`asm` 修饰符包含 `goto`（而在第一种形式中，不包含）。`asm` 关键字是一个 GNU 扩展。当编写使用 `-ansi` 和各种 `-std` 选项编译的代码时，使用 `__asm__` 而不是 `asm`（见[替代关键字](https://gcc.gnu.org/onlinedocs/gcc/Alternate-Keywords.html)）。

### 修饰符

- `volatile`
  扩展 `asm` 语句的典型用途是操作输入值以产生输出值。然而，`asm` 语句也可能产生副作用。如果样，可能需要使用 `volatile` 修饰符来禁用某些优化。见[Volatile](#Volatile)。
- `inline`
  如果使用 `inline` 修饰符，那么为了内联的目的，`asm` 语句的大小被认为是尽可能小的大小（见[asm 的大小](https://gcc.gnu.org/onlinedocs/gcc/Size-of-an-asm.html)）。
- `goto`
  这个限定符告知编译器，`asm` 语句可以执行跳转到 `Goto标签` 中列出的一个标签。见 2.10。

### 参数

- `汇编模板`
  这是一个字面字符串，是汇编程序代码的模板。它是一个固定的文本和指代输入、输出和 `goto` 参数的标记的组合。见[汇编模板](#汇编模板)。
- `输出操作数`
  一个以逗号分隔的列表，列出被汇编模板中的指令修改的 C 变量。允许有一个空列表。见[输出操作数](#输出操作数)。
- `输入操作数`
  一个以逗号分隔的列表，列出被汇编模板中的指令读取的 C 变量。允许有一个空列表。见[输入操作数](#输入操作数)。
- `Clobbers`
  一个以逗号分隔的列表，列出被汇编模板中的指令修改的寄存器或其他变量，除了被列为输出值的变量之外。允许有一个空列表。见 2.9。
- `Goto标签`
  当你使用 `asm` 的 `goto` 形式时，这部分包含汇编模板中代码可能跳转的所有 C 标签的列表。见 2.10。

  `asm` 语句不能跳转到其他 `asm` 语句，只能跳转到列出的 `Goto` 标签。GCC 的优化器不知道其他的跳转；因此在决定如何优化时不能考虑到它们。

输入、输出和 `goto` 操作数的总数被限制在 30 个。

### 备注

`asm` 语句允许在 C 代码中直接包含汇编指令。这可以帮助你在时间敏感的代码中最大限度地提高性能，或者访问 C 程序不容易获得的汇编指令。

请注意，扩展 `asm` 语句必须在一个函数内。只有基础 `asm` 可以在函数之外（见[基础汇编](#基础汇编——不带操作数的汇编程序指令)）。用 `naked` 属性声明的函数也需要基础 `asm`（见[函数属性](https://gcc.gnu.org/onlinedocs/gcc/Function-Attributes.html)）。

虽然 `asm` 的用途是多种多样的，但把 `asm` 语句看作是一系列将输入参数转换为输出参数的低级指令可能会有帮助。因此，一个简单的（如果不是特别有用）使用 `asm` 的 i386 的例子可能是这样的：

```cpp
int src = 1;
int dst;

asm ("mov %1, %0\n\t"
     "add $1, %0"
     : "=r" (dst)
     : "r" (src));

printf("%d\n", dst);
```

这段代码将`src`复制到`dst`，并对`dst`加 1。

### Volatile

GCC 的优化器有时会丢弃 `asm` 语句，如果认为不需要输出变量的话。此外，如果优化器认为代码将始终返回相同的结果（即在调用之间没有输入值变化），他们可能会将代码移出循环。使用 `volatile` 修饰符可以禁用这些优化。没有输出操作数的 `asm` 语句和 `asm goto` 语句隐含 `volatile`。

下面这段 i386 代码演示了一个不使用（或不需要）`volatile` 限定符的情况。如果它正在执行断言检查，这段代码使用 `asm` 来执行验证。否则，`dwRes` 就不会被任何代码引用。因此，优化器可以丢弃 `asm` 语句，这反过来又消除了对整个 `DoCheck` 例程的需要。通过在不需要时省略 `volatile` 修饰符，可以让优化器产生尽可能高效的代码。

```cpp
void DoCheck(uint32_t dwSomeValue)
{
    uint32_t dwRes;

    // Assumes dwSomeValue is not zero.
    asm ("bsfl %1,%0"
         : "=r" (dwRes)
         : "r" (dwSomeValue)
         : "cc");

    assert(dwRes > 3);
}
```

下一个例子展示了这样一种情况：优化器可以识别出输入（`dwSomeValue`）在函数执行过程中从未改变，因此可以将 `asm` 移到循环之外，以产生更有效的代码。同样，使用 `volatile` 修饰符可以禁止这种类型的优化。

```cpp
void do_print(uint32_t dwSomeValue)
{
    uint32_t dwRes;

    for (uint32_t x=0; x < 5; x++)
    {
        // Assumes dwSomeValue is not zero.
        asm ("bsfl %1,%0"
             : "=r" (dwRes)
             : "r" (dwSomeValue)
             : "cc");

        printf("%u: %u %u\n", x, dwSomeValue, dwRes);
    }
}
```

下面的例子演示了一个需要使用 `volatile` 修饰符的情况。它使用 x86 的 `rdtsc` 指令，该指令读取计算机的时间戳计数器。如果没有 `volatile` 修饰符，优化器可能会认为 `asm` 块会一直返回相同的值，因此优化掉第二个调用。

```cpp
uint64_t msr;

asm volatile ("rdtsc\n\t"          // Returns the time in EDX:EAX
              "shl $32, %%rdx\n\t" // Shift the upper bits left
              "or %%rdx, %0"       // 'Or' in the lower bits
              : "=a" (msr)
              :
              : "rdx");

printf("msr: %llx\n", msr);

// Do other work...

// Reprint the timestamp
asm volatile ("rdtsc\n\t"          // Returns the time in EDX:EAX
              "shl $32, %%rdx\n\t" // Shift the upper bits left
              "or %%rdx, %0"       // 'Or' in the lower bits
              : "=a" (msr)
              :
              : "rdx");

printf("msr: %llx\n", msr);
```

GCC 的优化器并不像前面例子中的非 `volatile` 代码那样对待这段代码。优化器不会把它移出循环，也不会在假设先前调用的结果仍然有效的情况下省略它。

请注意，编译器甚至可以将 `volatile asm` 指令相对于其他代码进行移动，包括跨跳转指令。例如，在许多目标架构上，有一个系统寄存器控制浮点运算的舍入模式。用 `volatile asm` 语句设置它，并不能可靠地工作，就像下面的 PowerPC 例子一样：

```cpp
asm volatile ("mtfsf 255, %0"
              :
              : "f" (fpenv));
sum = x + y;
```

编译器可能会将增加的内容移到 `volatile asm` 语句之前。为了使它像预期的那样工作，请在在随后的代码中引用一个变量，给`asm`添加一个人为的依赖关系，例如：

```cpp
asm volatile ("mtfsf 255, %1"
              : "=X" (sum)
              : "f" (fpenv));
sum = x + y;
```

在某些情况下，GCC 在优化时可能会重复（或删除重复的）你的汇编代码。如果你的 `asm` 代码定义了符号或标签，这可能导致在编译过程中出现意想不到的重复符号错误。使用 `%=`（见[汇编模板](#汇编模板)）可能有助于解决这个问题。

### 汇编模板

汇编模板是一个包含汇编指令的字面字符串。编译器替换模板中指代输入、输出和 `goto` 标签的标记，然后将结果字符串输出给汇编器。这个字符串可以包含任何被汇编器识别的指令（instructions），包括汇编器指令（directives）。GCC 并不解析汇编指令本身，也不知道它们是什么意思，甚至不知道它们是否是有效的汇编器输入。然而，它确实计算了这些语句（见[asm 的大小](https://gcc.gnu.org/onlinedocs/gcc/Size-of-an-asm.html)）。

可以将多条汇编指令放在一个 `asm` 字符串中，用系统的汇编代码中通常使用的字符隔开。一个在大多数地方都有效的组合是用换行符来分隔行，再加上一个制表符来移动到指令域（写成`\n\t`）。一些汇编器允许用分号作为行的分隔符。然而，请注意，有些汇编方言使用分号来开始注释。

即使使用了 `volatile` 限定符，也不要期望一连串的 `asm` 语句在编译后能保持完全连续。如果某些指令需要在输出中保持连续，可以把它们放在一个一条多指令 `asm` 语句中。

在不使用输入/输出操作数的情况下从 C 程序中访问数据（比如直接从汇编器模板中使用全局符号），可能无法达到预期效果。同样，直接从汇编器模板中调用函数需要对目标汇编器和 ABI 有详细的了解。

由于 GCC 没有解析汇编模板，所以它对引用的任何符号都不可见。这可能会导致 GCC 将这些符号作为未引用的符号丢弃，除非它们也被列为输入、输出或 `goto` 操作数。

#### 特殊格式字符串

除了输入、输出和 `goto` 操作数所描述的标记外，这些标记在汇编模板中具有特殊的含义：

- `%%`
  在汇编代码中输出一个单一的 `%`。
- `%=`
  输出一个数字，这个数字对整个编译中 `asm` 语句的每个实例都是唯一的。这个选项在创建本地标签并在生成多个汇编指令的单一模板中多次引用它们时很有用。
- `%{`
- `%|`
- `%}`
  将 `{`、`|` 和 `}` 字符分别输出到汇编程序代码中。当未转义时，这些字符具有特殊的意义，表示多种汇编器方言，如下所述。

#### 汇编模板中的多种汇编方言

在一些平台，例如 x86 上，GCC 支持多种汇编方言。`-masm` 选项控制 GCC 对于内联汇编使用哪种方言。关于 `-masm` 选项的目标架构特定的文档[^2]包含了支持的方言列表，以及在没有指定该选项时的默认方言。了解这些信息很重要，因为在使用某种方言编译时能正常工作的汇编代码，在使用另一种方言编译时可能会失败。见[x86 选项](https://gcc.gnu.org/onlinedocs/gcc/x86-Options.html)。

如果你的代码需要支持多种汇编方言（例如，如果你编写的公共头文件需要支持多种编译选项），请使用这种形式的结构：

```cpp
{ 方言0 | 方言1 | 方言2... }
```

当使用方言 #0 编译代码时，该结构输出 `方言0`，方言 #1 则输出 `方言1`，等等。如果大括号中的备选方案少于编译器所支持的方言数量，那么该结构将不输出任何信息。

例如，如果一个 x86 编译器支持两种方言（`att`，`intel`），那么一个像这样的汇编模板：

```cpp
"bt{l %[Offset],%[Base] | %[Base],%[Offset]}; jc %l2"
```

相当于两者之一：

```cpp
"btl %[Offset],%[Base] ; jc %l2" /* att 方言   */
"bt %[Base],%[Offset]; jc %l2"   /* intel 方言 */
```

使用同样的编译器，下面的代码：

```cpp
"xchg{l}\t{%%}ebx, %1"
```

相当于两者之一：

```cpp
"xchgl\t%%ebx, %1" /* att 方言   */
"xchg\tebx, %1"    /* intel 方言 */
```

编译器不支持方言替换的嵌套。

### 输出操作数

`asm` 语句有零个或多个输出操作数，表示被汇编程序代码修改的 C 变量的名称。

在这个 i386 的例子中，`old`（在模板字符串中作为 `%0`）和 `*Base`（作为 `%1`）是输出，`Offset`（`%2`）是输入。

```cpp
bool old;

__asm__ ("btsl %2, %1\n\t" // Turn on zero-based bit #Offset in Base.
         "sbb %0, %0"      // Use the CF to calculate old.
         : "=r" (old), "+rm" (*Base)
         : "Ir" (Offset)
         : "cc");

return old;
```

操作数由逗号分隔。每个操作数都有如下的格式：

```
[asm 符号名称] 约束 (C 变量名称)
```

#### asm 符号名称

为操作数指定一个符号名称。在汇编模板中引用这个名字时，要把它放在方括号里（即 `%[Value]`）。该名称的范围是包含该定义的 `asm` 语句。任何有效的 C 变量名称都是可以接受的，包括周围代码中已经定义的名称。在同一个 `asm` 语句中，没有两个操作数可以使用相同的符号名。

当不使用 `asm 符号名称` 时，使用操作数在汇编模板的操作数列表中的位置（从零开始）。例如，如果有三个输出操作数，在模板中使用 `%0` 来指代第一个，`%1` 指代第二个，而 `%2` 指代第三个。

#### 约束

一个字符串常量，指定对操作数位置的约束；详见[对 asm 操作数的约束](#对-asm-操作数的约束)。

输出约束必须以 `=`（变量覆盖现有值）或 `+`（读和写）开始。当使用=时，不要假设该位置包含进入 `asm` 的现有值，除非操作数与输入绑定；见[输入操作数](#输入操作数)。

在前缀之后，必须有一个或多个额外的约束条件（见[对 asm 操作数的约束](#对-asm-操作数的约束)）来描述值的位置。常见的约束包括代表寄存器的 `r` 和代表内存的 `m`。当你列出多个可能的位置时（例如，`=rm`），编译器会根据当前的环境选择最有效的一个。如果你在 `asm` 语句允许的范围内列出尽可能多的备用位置，你就允许优化器产生尽可能好的代码。如果你必须使用一个特定的寄存器，但是你的机器约束并没有提供足够的控制来选择你想要的特定寄存器，那么局部寄存器变量可以提供一个解决方案（见局部寄存器变量）。

#### C 变量名

指定一个 C 语言左值表达式来保存输出，通常是一个变量名。括号是语法的一个必要部分。

####

当编译器选择用来表示输出操作数的寄存器时，它不会使用任何一个 clobbered 寄存器（见[Clobbers 和临时寄存器](#Clobbers-和临时寄存器)）。

输出操作数表达式必须是左值。编译器不会检查操作数的数据类型对正在执行的指令是否合理。对于不能直接寻址的输出表达式（例如一个位域），约束条件必须允许一个寄存器。在这种情况下，GCC 使用寄存器作为 `asm` 的输出，然后将该寄存器存储到输出中。使用 `+` 约束修饰符的操作数算作两个操作数（即既是输入又是输出），每个 `asm` 语句最多有 30 个操作数。

在所有不得与输入重叠的输出操作数上使用 `&` 约束修饰符[^3]（见修饰符）。否则，GCC 可能会将输出操作数分配到与不相关的输入操作数相同的寄存器中，其假设是汇编程序代码在产生输出之前消耗其输入。如果汇编程序代码实际上由一条以上的指令组成，这个假设可能是错误的。

如果一个输出参数（`a`）允许有寄存器约束，而另一个输出参数（`b`）允许有内存约束，也会出现同样的问题。GCC 生成的访问 `b` 中内存地址的代码可能包含可能被 `a` 共享的寄存器，而 GCC 认为这些寄存器是 `asm` 的输入。如上所述，GCC 假定这种输入寄存器在任何输出被写入之前就被消耗了。如果 `asm` 语句在使用 `b` 之前写入 `a`，这个假设可能会导致不正确的行为。将 `&` 修饰符和 `a` 的寄存器约束结合起来，可以确保修改 `a` 不会影响 `b` 引用的地址。

`asm` 支持操作数的修饰符（例如 `%k2` 而不是简单的 `%2`）。通常，这些修饰符与硬件有关。x86 支持的修饰符的列表见[x86 操作数修饰符][#x86-操作数修饰符]。

如果 `asm` 后面的 C 代码没有使用任何输出操作数，那么对 `asm` 语句使用 `volatile`，以防止优化器将 `asm` 语句作为非必要的东西抛弃（见[Volatile](#Volatile)）。

这段代码没有使用可选的 `asm符号名称`。因此，它引用第一个输出操作数为 `%0`（如果有第二个，它将是 `%1`，等等）。第一个输入操作数比最后一个输出操作数的数字大一个。在这个 i386 的例子中，这使得 `Mask` 被引用为 `%1`。

```cpp
uint32_t Mask = 1234;
uint32_t Index;

asm ("bsfl %1, %0"
     : "=r" (Index)
     : "r" (Mask)
     : "cc");
```

这段代码覆盖了变量 `Index`（`=`），将数值放在了一个寄存器（`r`）中。使用通用的 `r` 约束而不是特定寄存器的约束，允许编译器选择要使用的寄存器，这可能会导致更有效的代码。如果一条汇编指令需要一个特定的寄存器，这或许是不可能的。

下面的 i386 例子使用 `asm符号名称` 语法。它产生的结果与上面的代码相同，但有些人可能认为它更易读或更易维护，因为在添加或删除操作数时不需要对索引号进行重新排序。在这个例子中，`aIndex` 和 `aMask` 这两个名字只是为了强调哪个名字在哪里被使用。重复使用 `Index` 和 `Mask` 这两个名字是可以接受的。

```cpp
uint32_t Mask = 1234;
uint32_t Index;

asm ("bsfl %[aMask], %[aIndex]"
     : [aIndex] "=r" (Index)
     : [aMask] "r" (Mask)
     : "cc");
```

下面是一些关于输出操作数的例子。

```cpp
uint32_t c = 1;
uint32_t d;
uint32_t *e = &c;

asm ("mov %[e], %[d]"
     : [d] "=rm" (d)
     : [e] "rm" (*e));
```

这里，`d` 可以在一个寄存器中，也可以在内存中。由于编译器可能已经有了寄存器中 `e` 所指向的 `uint32_t` 位置的当前值，你可以通过指定这两个约束来使它为 `d` 选择最佳位置。

### 输入操作数

输入操作数使 C 语言变量和表达式的值可用于汇编代码。

操作数由逗号分隔。每个操作数都有这样的格式：

```cpp
[asm符号名称] 约束 (C表达式)
```

#### asm符号名称

为操作数指定一个符号名称。在汇编模板中引用这个名字时，要把它放在方括号里（即`%[Value]`）。该名称的范围是包含该定义的 `asm` 语句。任何有效的 C 变量名称都是可以接受的，包括周围代码中已经定义的名称。在同一个 `asm` 语句中，没有两个操作数可以使用相同的符号名。

当不使用 `asm符号名称` 时，使用操作数在汇编模板的操作数列表中的位置（从零开始）。例如，如果有两个输出操作数和三个输入操作数，在模板中使用 `%2` 来指代第一个输入操作数，`%3` 指代第二个，而 `%4` 指代第三个。

#### 约束

一个字符串常数，指定对操作数位置的约束；详见[对 asm 操作数的约束](#对-asm-操作数的约束)。输入约束字符串不能以 `=` 或 `+` 开头。当你列出多个可能的位置时（例如，`"irm"`），编译器会根据当前环境选择最有效的位置。如果你必须使用一个特定的寄存器，但是你的机器约束没有提供足够的控制来选择你想要的特定寄存器，那么局部寄存器变量可以提供一个解决方案（见[局部寄存器变量](https://gcc.gnu.org/onlinedocs/gcc/Local-Register-Variables.html)）。

输入约束也可以是数字（例如，`"0"`）。这表明指定的输入必须与输出约束在输出约束列表中的索引处（从零开始）处于同一位置。当对输出操作数使用 `asm符号名称` 语法时，你可以使用这些名称（用括号 `[]` 括起来），而不是数字。

#### C 表达式

这是作为输入传递给 `asm` 语句的 C 变量或表达式。括号是语法的一个必要部分。

####

当编译器选择用来表示输入操作数的寄存器时，它不会使用任何一个 clobbered 寄存器（见[Clobbers 和临时寄存器](#Clobbers-和临时寄存器)）。

如果没有输出操作数，但有输入操作数，则在输出操作数的位置连续放置两个冒号：

```cpp
__asm__ ("some instructions"
         : /* No outputs. */
         : "r" (Offset / 8));
```

**警告：** 不要修改只输入的操作数的内容（与输出绑定的输入除外）。编译器假定在退出 `asm` 语句时，这些操作数包含与执行语句前相同的值。不可能使用 clobbers 来通知编译器这些输入的值在变化。一个常见的解决方法是将变化的输入变量与一个永远不会被使用的输出变量联系起来。但是请注意，如果 `asm` 语句后面的代码没有使用任何输出操作数，GCC 优化器可能会将 `asm` 语句作为不需要的东西丢弃（见[Volatile](#Volatile)）。

`asm` 支持操作数的修饰符（例如 `%k2` 而不是简单的 `%2`）。通常，这些修饰符与硬件有关。x86 支持的修饰符的列表见[x86 操作数修饰符][#x86-操作数修饰符]。

在这个使用虚构的 `combine` 指令的例子中，输入操作数 1 的约束 `"0"` 表示它必须与输出操作数 0 占据相同的位置。只有输入操作数可以在约束中使用数字，而且它们必须各自指向一个输出操作数。只有约束中的数字（或符号化的汇编器名称）才能保证一个操作数与另一个操作数在同一位置。仅仅是 `foo` 是两个操作数的值，并不足以保证它们在生成的汇编代码中处于相同的位置。

```cpp
asm ("combine %2, %0"
     : "=r" (foo)
     : "0" (foo), "g" (bar));
```

下面是一个使用符号名称的例子。

```cpp
asm ("cmoveq %1, %2, %[result]"
     : [result] "=r"(result)
     : "r" (test), "r" (new), "[result]" (old));
```

### Clobbers 和临时寄存器

虽然编译器知道输出操作数中列出的条目的变化，但内联 `asm` 代码可能修改的不仅仅是输出。例如，计算可能需要额外的寄存器，或者处理器可能会覆盖一个寄存器作为一个特定汇编指令的副作用。为了通知编译器这些变化，将它们列在 clobber 列表中。clobber 列表项要么是寄存器名称，要么是特殊的 clobbers（下面列出）。每个 clobber 列表项是一个字符串常数，用双引号括起来，用逗号隔开。

Clobber 描述不得以任何方式与输入或输出操作数重合。例如，当在 clobber 列表中列出一个寄存器类时，不能有一个操作数来描述该寄存器的一个成员。被声明为处于特定寄存器中的变量（见显式寄存器变量）和作为 `asm` 输入或输出操作数使用的变量必须在 clobber 描述中没有提到任何部分。特别是，如果不把输入操作数指定为输出操作数，就没有办法指定输入操作数被修改。

当编译器选择使用哪些寄存器来表示输入和输出操作数时，它不会使用任何 clobbered 的寄存器。因此，clobbered 寄存器可以在汇编代码中用于任何用途。

另一个限制是，clobber 列表不应该包含栈指针寄存器。这是因为编译器要求栈指针的值在 `asm` 语句之后与进入该语句时相同。然而，以前的 GCC 版本没有执行这一规则，允许栈指针出现在列表中，但语义不明确。这种行为已被废弃，在未来的 GCC 版本中，列出栈指针可能成为一个错误。

下面是一个展示了 clobbered 寄存器使用的 VAX 实例：

```cpp
asm volatile ("movc3 %0, %1, %2"
              : /* No outputs. */
              : "g" (from), "g" (to), "g" (count)
              : "r0", "r1", "r2", "r3", "r4", "r5", "memory");
```

此外，还有两个特殊的 clobber 参数：

- `"cc"`
  `"cc"` clobber 表示汇编程序代码修改了标志寄存器。在一些机器上，GCC 将条件代码表示为一个特定的硬件寄存器；`"cc"` 用于命名这个寄存器。在其他机器上，条件代码的处理是不同的，指定 `"cc"` 没有影响。但是，无论目标架构是什么，它都是有效的。
- `"memory"`
  `"memory"` clobber 告知编译器，汇编代码对输入和输出操作数中所列项目以外的项目进行内存读取或写入（例如，访问输入参数之一所指向的内存）。为了确保内存包含正确的值，GCC 可能需要在执行 `asm` 之前将特定的寄存器值刷入内存。此外，编译器并不假定在 `asm` 之前从内存中读出的任何数值在该 `asm` 之后保持不变；它会根据需要重新加载它们。使用 `"memory"` clobber 以有效地形成编译器的内存读/写屏障。

  请注意，这个 clobber 并不能阻止处理器在 `asm` 语句之后进行预测性读取[^4]，为了防止这一点，需要使用处理器特定的[^5]存储器屏蔽指令[^6]。

将寄存器刷入内存会影响性能，对于时间敏感的代码来说可能是个问题。你可以向 GCC 提供更好的信息来避免这种情况，如下面的例子所示。至少，别名规则[^7]允许 GCC 知道哪些内存不需要被刷新。

下面是一条虚构的平方和指令，它接收两个指向内存中浮点值的指针并产生一个浮点寄存器输出。注意 `x` 和 `y` 在 `asm` 参数中都出现了两次，一次是指定访问的内存，一次是指定 `asm` 使用的基础寄存器。你这样做通常不会浪费一个寄存器，因为 GCC 可以将同一个寄存器用于两种用途。然而，在这个 `asm` 中同时使用 `%1` 和 `%3` 来表示 `x`，并期望它们是一样的，这是愚蠢的行为。事实上，`%3` 很可能不是一个寄存器。它可能是对 `x` 所指向的对象的一个符号化的内存引用。

```cpp
asm ("sumsq %0, %1, %2"
     : "+f" (result)
     : "r" (x), "r" (y), "m" (*x), "m" (*y));
```

下面是一条虚构的 `*z++ = *x++ * *y++` 指令。注意，`x`、`y` 和 `z` 指针寄存器必须被指定为输入/输出，因为 `asm` 会修改它们。

```cpp
asm ("vecmul %0, %1, %2"
     : "+r" (z), "+r" (x), "+r" (y), "=m" (*z)
     : "m" (*x), "m" (*y));
```

一个 x86 的例子，字符串内存参数的长度未知：

```cpp
asm ("repne scasb"
     : "=c" (count), "+D" (p)
     : "m" (*(const char (*)[]) p), "0" (-1), "a" (0));
```

如果已知上面只读一个十字节的数组，那么可以改用一个内存输入，例如：`"m"(*(const char (*)[10])p)`。

下面是一个用汇编实现的 PowerPC 矢量表的例子，其中包括矢量和条件代码 clobbers，以及一些初始化的偏移寄存器，这些都没有被 `asm` 改变。

```cpp
void dscal (size_t n, double *x, double alpha)
{
    asm ("/* lots of asm here */"
         : "+m" (*(double (*)[n]) x), "+&r" (n), "+b" (x)
         : "d" (alpha), "b" (32), "b" (48), "b" (64),
           "b" (80), "b" (96), "b" (112)
         : "cr0",
           "vs32", "vs33", "vs34", "vs35",
           "vs36", "vs37", "vs38", "vs39",
           "vs40", "vs41", "vs42", "vs43",
           "vs44", "vs45", "vs46", "vs47");
}
```

与其通过 clobbers 分配固定寄存器来为 `asm` 语句提供临时寄存器，不如定义一个变量并使其成为 early-clobber 输出，就像下面例子中的 `a2` 和 `a3` 一样。这给了编译器寄存器分配器更多的自由。你也可以定义一个变量，让它成为一个与输入绑定的输出，如 `a0` 和 `a1`，分别与 `ap` 和 `lda` 绑定。当然，对于绑定的输出，你的 `asm` 在修改了输出寄存器后不能使用输入值，因为它们是同一个寄存器。更重要的是，如果你省略了输出上的 early-clobber，如果 GCC 能够证明它们在进入 `asm` 时具有相同的值，那么 GCC 就有可能将相同的寄存器分配给另一个输入。这就是为什么 `a1` 有一个 early-clobber。可以想象，它的并列输入 `lda` 的值是 16，如果没有 early-clobber，就会和 `%11` 共享同一个寄存器。另一方面，`ap` 不可能与其他任何输入相同，所以 `a0` 上不需要一个 early-clobber。在这种情况下，它也是不可取的。在 `a0` 上的一个 early-clobber 会导致 GCC 为 `"m"(*(const double (*)[])ap)` 输入分配一个单独的寄存器。请注意，将一个输入与一个输出捆绑在一起是建立一个被 `asm` 语句修改的初始化临时寄存器的方法。没有与输出绑定的输入被 GCC 认为是没有变化的，例如下面的 `"b"`（16）将 `%11` 设置为 16，如果刚好需要 16 这个值，GCC 可能会在后面的代码中使用这个寄存器。你甚至可以用一个正常的 `asm` 输出来做一个 scratch，如果所有可能共享同一个寄存器的输入在使用 scratch 之前就被消耗掉了。除了 GCC 对 `asm` 参数数量的限制外，被 `asm` 语句 clobber 的 VSX 寄存器可以使用这种技术。

```cpp
static void dgemv_kernel_4x4 (long n, const double *ap, long lda,
                              const double *x, double *y,
                              double alpha)
{
    double *a0;
    double *a1;
    double *a2;
    double *a3;

    __asm__ (/* lots of asm here */
             "#n=%1 ap=%8=%12 lda=%13 x=%7=%10 y=%0=%2 alpha=%9 o16=%11\n"
             "#a0=%3 a1=%4 a2=%5 a3=%6"
             : "+m" (*(double (*)[n]) y),
               "+&r" (n), & // 1
               "+b" (y), & // 2
               "=b" (a0), & // 3
               "=&b" (a1), & // 4
               "=&b" (a2), & // 5
               "=&b" (a3) & // 6
             : "m" (*(const double (*)[n]) x),
               "m" (*(const double (*)[]) ap),
               "d" (alpha), & // 9
               "r" (x), &  & // 10
               "b" (16), & // 11
               "3" (ap), & // 12
               "4" (lda) & // 13
             : "cr0",
               "vs32", "vs33", "vs34", "vs35", "vs36",
               "vs37", "vs40", "vs41", "vs42", "vs43",
               "vs44", "vs45", "vs46", "vs47");
}
```

### Goto 标签

`asm goto` 允许汇编代码跳到一个或多个 C 标签。`asm goto` 语句中的 `Goto 标签` 部分包含一个以逗号分隔的所有 C 标签的列表，汇编代码可以跳到这些标签。GCC 假定 `asm` 的执行会跳到下一条语句（如果不是这样，可以考虑在 `asm` 语句后使用 `__builtin_unreachable` 本征）。`asm goto` 的优化可以通过使用 `hot` 标签和 `cold` 标签属性来改善（见[标签属性](https://gcc.gnu.org/onlinedocs/gcc/Label-Attributes.html)）。

如果汇编程序代码确实修改了一些东西，使用 `"memory"` clobber 来强制优化器将所有的寄存器值刷入内存中，并在 `asm` 语句之后必要时重新加载它们。

要在汇编器模板中引用一个标签，请在它前面加上 `%l`（小写 L），再加上它在 `Goto 标签` 中的位置（从零开始）和输入和输出操作数。带有约束修饰符 `+` 的输出操作数被算作两个操作数，因为它被认为是一个输出和一个输入操作数。例如，如果 `asm` 有三个输入，一个带有约束修饰符 `+` 的输出操作数和一个带有约束修饰符 `=` 的输出操作数，并且引用了两个标签，将第一个标签称为 `%l6`，第二个称为 `%l7`。

另外，你也可以使用括号内的实际 C 标签名称来引用标签。例如，要引用一个名为 `carry` 的标签，你可以使用 `%l[carry]`。使用这种方法时，该标签仍然必须被列在 `Goto 标签` 部分。最好使用标签的命名引用，因为在这种情况下，你可以避免计算输入和输出操作数，以及用约束修饰符 `+` 对输出操作数进行特殊处理。

下面是 i386 中关于 `asm goto` 的一个例子：

```cpp
asm goto ("btl %1, %0\n\t"
          "jc %l2"
          : /* No outputs. */
          : "r" (p1), "r" (p2)
          : "cc"
          : carry);

return 0;

carry:
return 1;
```

下面的例子展示了了一个使用存储器 clobber 的 `asm goto`。

```cpp
int frob(int x)
{
    int y;
    asm goto ("frob %%r5, %1; jc %l[error]; mov (%2), %%r5"
              : /* No outputs. */
              : "r"(x), "r"(&y)
              : "r5", "memory"
              : error);
    return y;
error:
    return -1;
}
```

下面的例子展示了一个使用输出的 `asm goto`。

```cpp
int foo(int count)
{
    asm goto ("dec %0; jb %l[stop]"
              : "+r" (count)
              :
              :
              : stop);
    return count;
stop:
    return 0;
}
```

下面的例子展示了一个 `asm goto`，它只在 `asm goto` 内部的一条路径上设置了一个输出。使用约束修饰符 `=` 而不是 `+` 将是错误的，因为 `factor` 被用于 `asm goto` 的所有路径。

```cpp
int foo(int inp)
{
    int factor = 0;
    asm goto ("cmp %1, 10; jb %l[lab]; mov 2, %0"
              : "+r" (factor)
              : "r" (inp)
              :
              : lab);
lab:
    return inp * factor; /* return 2 * inp or 0 if inp < 10 */
}
```

### x86 操作数修饰符

在扩展 `asm` 语句的汇编模板中，对输入、输出和 `goto` 操作数的引用，可以使用修饰符来影响操作数在输出到汇编器的代码中的格式。例如，下面的代码对 x86 使用了 `h` 和 `b` 修饰符：

```cpp
uint16_t num;
asm volatile ("xchg %h0, %b0" : "+a" (num));
```

这些修饰符生成了如下的汇编代码：

```assembly
xchg %ah, %al
```

本讨论的其余部分使用以下代码来说明问题。

int main()
{

top:

    asm volatile goto ("some assembler instructions here"
                       : /* No outputs. */
                       : "q" (iInt), "X" (sizeof(unsigned char) + 1), "i" (42)
                       : /* No clobbers. */
                       : top);
}

### x86 浮点 asm 操作数

在 x86 目标架构上，在 `asm` 的操作数中有几个关于类栈寄存器的使用规则。这些规则只适用于作为类栈寄存器的操作数。

1. 给出一组在 `asm` 中停止运作的输入寄存器，有必要知道哪些是被 `asm` 隐式弹出的，哪些必须被 GCC 显式弹出。

被 `asm` 隐式弹出的输入寄存器必须被显式弹出，除非它被限制为与输出操作数匹配。

2. 对于任何被 `asm` 隐式弹出的输入寄存器，有必要知道如何调整栈以补偿弹出的情况。如果任何非弹出的输入比隐式弹出的寄存器更接近 reg-stack 的顶部，那么就不可能知道栈的样子–不清楚栈的其他部分是如何“滑上去”的。所有隐式弹出的输入寄存器必须比任何没有隐式弹出的输入更接近 reg-stack 的顶部。

如果一个输入在 `asm` 中停止运作，编译器有可能使用该输入寄存器进行输出重载。考虑一下这个例子：

```cpp
asm ("foo" : "=t" (a) : "f" (b));
```

这段代码说，输入 b 没有被 `asm` 弹出，而 `asm` 把一个结果推到了 reg-stack 上，也就是说，`asm` 之后的栈比之前的栈要深一个。但是，reload 有可能认为它可以将同一个寄存器用于输入和输出。

为了防止这种情况发生，如果任何输入操作数使用 `f` 约束，所有的输出寄存器约束必须使用 early-clobber 修饰符。

上面的例子被正确地写成了：

```cpp
asm ("foo" : "=&t" (a) : "f" (b));
```

3. 有些操作数需要在栈中的特定位置。所有的输出操作数都属于这一类，除非你在约束中指出，否则 GCC 没有办法知道输出出现在哪个寄存器中。输出操作数必须在 `asm` 后明确指出输出出现在哪个寄存器中。`=f` 是不允许的：操作数约束必须选择一个具有单个寄存器的类。

4. 输出操作数不能“插入”现有的栈寄存器之间。因为没有 387 操作码使用读/写操作数，所有的输出操作数在 `asm` 之前都是死的，并由 `asm` 推送。除了在 reg-stack 的顶部推送，在其他地方推送是没有意义的。

5. 一些 asm 语句可能需要额外的栈空间用于内部计算。这可以通过删减与输入和输出无关的栈寄存器来保证。

这个 `asm` 需要一个输入，这个输入是内部弹出的，并产生两个输出：

```cpp
asm ("fsincos" : "=t" (cos), "=u" (sin) : "0" (inp));
```

这个 `asm` 需要两个输入，由 `fyl2xp1` 操作码弹出，并以一个输出代替它们。`st(1)` 代码对于编译器知道 `fyl2xp1` 弹出两个输入是必要的。

```cpp
asm ("fyl2xp1" : "=t" (result) : "0" (x), "u" (y) : "st(1)");
```

## 对 asm 操作数的约束

下面是关于可以对 asm 操作数使用哪些约束字母的具体细节。约束可以说一个操作数是否可以在一个寄存器中，以及哪些类型的寄存器；操作数是否可以是一个内存引用，以及哪些类型的地址；操作数是否可以是一个立即数，以及它可能有哪些值。约束也可以要求两个操作数相匹配。在内联 `asm` 的操作数中不允许有副作用，除非使用 `>` 或 `<` 约束，因为不能保证副作用在一条可以更新寻址寄存器的指令中正好发生一次。

### 简单约束

最简单的约束是一个充满字母的字符串，每个字母描述一种允许的操作数。下面是允许的字母：

- 空白字符
  空白字符被忽略，可以在任何位置插入，除了第一个位置。这使得不同操作数的每个备选方案在机器描述中都能在视觉上保持一致，即使它们有不同数量的约束和修饰符。
- `m`
  一个具有机器一般支持的任何种类的地址的内存操作数。注意，用于一般内存约束的字母可以由后端使用 `TARGET_MEM_CONSTRAINT` 宏重新定义。
- `o`
  一个具有可偏移[^10]地址的内存操作数。这意味着可以在地址上添加一个小的整数（实际上是操作数的字节宽度，由其机器模式决定），其结果也是一个有效的内存地址。例如，一个常数的地址是可偏移的；一个寄存器和常数之和的地址也是可偏移的（只要一个稍大的常数也在机器支持的地址偏移范围内）；但一个自动递增或自动递减的地址是不可偏移的。更复杂的间接/索引地址可能是可偏移的，也可能不是，取决于机器支持的其他寻址模式。

  注意，在一个可以被另一个操作数匹配的输出操作数中，约束字母 `o` 只有在伴随着 `<`（如果目标机有预增寻址）和 `>`（如果目标机有预增寻址）时才有效。

- `V`
  一个不可偏移的内存操作数。换句话说，任何符合 `m` 约束但不符合 `o` 约束的操作数。
- `<`
  一个具有自动递减寻址的内存操作数（无论是前递减还是后递减）。在内联 `asm` 中，只有当操作数在一条可以处理副作用的指令中正好使用一次时，才允许这种约束条件。在内联 `asm` 模式中完全不使用约束字符串中的 `|<|` 的操作数或在多条指令中使用它是无效的，因为副作用不会被执行或会被执行一次以上。此外，在某些目标上，约束字符串中带有 `<` 的操作数必须伴有特殊的指令后缀，如 PowerPC 上的 `%U0` 指令后缀或 IA-64 上的 `%P0`。
- `>`
  一个具有自动递增寻址的内存操作数（无论是前递增还是后递增）。在内联 `asm` 中，适用与 `<` 相同的限制。
- `r`
  一个寄存器操作数，只要它是在一个通用寄存器中。
- `i`
  一个立即整数操作数（有常量值的）。这包括符号常数，其值只有在汇编期或以后才会知道。
- `n`
  一个已知数值的立即整数操作数。许多系统不能支持小于一个字宽的操作数的汇编期常量。这些操作数的约束应该使用 `n` 而不是 `i`。
- `I`、`J`、`K`……`P`
  从 `I` 到 `P` 内的字母视机器的情况而定义，以允许在指定范围内有明确的整数值的立即整数操作数。例如，在 68000[^11]上，`I` 被定义为代表 1 到 8 的数值范围。在移位指令中作为允许的移位数的范围。
- `E`
  一个立即浮点操作数（表达式代码 `const_double`），但只有在目标浮点格式与主机（编译器运行在其上）的浮点格式相同时才允许使用。
- `F`
  一个立即浮点操作数（表达式代码 `const_double` 或 `const_vector`）。
- `G`、`H`
  `G` 和 `H` 可以以一种依赖于机器的方式定义，以允许在特定数值范围内的立即浮点操作数。
- `s`
  允许一个立即整数操作数，其值不是一个明确的整数。

  这可能看起来很奇怪：如果一个指令允许一个在编译时不知道值的常数操作数，那么它当然必须允许任何已知值。那么，为什么使用 `s` 而不是 `i` 呢？有时它能使代码生成得更好。

  例如，在 68000 的全字指令中，可以使用一个立即操作数；但是如果立即值在 −128 和 127 之间，将该值加载到寄存器中并使用该寄存器可以得到更好的代码。这是因为加载到寄存器的过程可以用 `moveq` 指令完成。我们通过定义字母 K 表示“−128 到 127 范围外的任何整数”，然后在操作数约束中指定 `Ks` 来安排这种情况的发生。

- `g`
  任何寄存器、内存或立即整数操作数都是允许的，非通用寄存器除外。
- `X`
  任何操作数都是允许的。
- `0`、`1`、`2`……`9`
  允许与指定的操作数相匹配的操作数。如果一个数字与字母一起使用在同一个备选方案中，数字应该放在最后。

  这个数字允许多于一个数字。如果连续遇到多个数字，它们会被解释为一个十进制的整数。很少会发生歧义，因为迄今为止，`10` 被解释为与操作数 1 或操作数 0 相匹配是不可取的。

  此即所谓的匹配约束，其真正含义是，汇编器只有一个操作数，它填补了 `asm` 所区分的两个角色。例如，一条加法指令使用两个输入操作数和一个输出操作数，但在大多数 CISC 机器上，加法指令实际上只有两个操作数，其中一个是输入-输出操作数：

```cpp
addl #35,r12
```

在这些情况下使用匹配约束。更确切地说，匹配的两个操作数必须包括一个只输入的操作数和一个只输出的操作数。此外，该数字必须比约束中使用它的操作数的数字小。

- `p`
  允许操作数是一个有效的内存地址。这适用于“load address”和“push address”指令。

  在约束中的 `p` 必须伴随着 `address_operand` 作为 `match_operand` 的谓词。这个谓词将 `match_operand` 中指定的模式解释为地址有效的内存引用的模式。

- 其他字母
  其他字母可以根据机器的情况来定义，以代表特定类别的寄存器或其他任意的操作数类型。`d`、`a` 和 `f` 在 68000/68020 上被定义为代表数据、地址和浮点寄存器。

### 多重限制

有时一条指令有多组可能的操作数。例如，在 68000 指令中，逻辑或指令可以将寄存器或即期值合并到内存中，也可以将任何类型的操作数合并到寄存器中；但不能将一个内存位置合并到另一个内存位置。

这些约束被表示为多种选择。一个替代方案可以用每个操作数的一系列字母来描述。一个操作数的总约束是由第一个字母和一个逗号，第二个字母和一个逗号组成，依次类推，直到最后一个操作数。一条指令的所有操作数都必须有相同数量的选项。

所以 68000 的逻辑或的第一个选项可以写成 `"+m"(output): "ir"(input)`。第二条可以写成 `"+r"(output): "irm"(input)`。然而，两个内存位置不能在一条指令中使用的事实使我们不能简单地使用 `"+rm"(output): "irm"(input)`。使用多重替代指令，可以写成 `"+m,r"(output): "ir,irm"(input)`。这就向编译器描述了所有可用的替代方案，允许它在当前条件下选择最有效的方案。

模板中无法确定选择了哪种方案。然而，可以用 builtins（如 `__builtin_constant_p`）来包装 `asm` 语句，以达到预期的效果。

### 操作数修饰符

以下是操作数修饰符：

- `=`
  表明该操作数被指令写入：此前的值被丢弃，被新的数据取代。
- `+`
  表明该操作数既被指令读取也被指令写入。

  当编译器妥善安置操作数以满足约束条件时，它需要知道哪些操作数是指令读取的，哪些是写入的。`=` 标识一个只被写入的操作数；`+` 标识一个既被读又被写的操作数；所有其他操作数被假定为只被读。

  如果你在约束中指定 `=` 或 `+`，你就把它放在约束字符串的第一个字符中。

- `&`
  表明（在一个特定的备选方案中）该操作数是一个 early-clobber 操作数，在指令使用完输入操作数之前被写入。因此，这个操作数不能位于指令读取的寄存器中，也不能作为任何内存地址的一部分。

  `&` 只适用于它被写入的选项。在有多个选项的约束中，有时一个选项需要 `&`，而其他选项不需要。例如，见 68000 的 `movdf` 指令。

  如果一个被指令读取的操作数在 early-clobber 被写入之前只作为输入使用，那么该操作数就可以被绑定到一个 `early-clobber` 操作数上。添加这种形式的替代物通常可以使 GCC 在只有一些读操作数可以被早期循环器影响时产生更好的代码。例如，见 ARM 的 `mulsi3` 指令。

  此外，如果 early-clobber 操作数也是一个读/写操作数，那么该操作数只有在被使用后才被写入。

  `&` 并不能避免写 `=` 或 `+` 的需要。由于 early-clobber 操作数总是被写入的，一个只读的 early-clobber 操作数是不符合格式的，会被编译器拒绝。

- `%`
  声明该指令对这个操作数和后面的操作数是互换的。这意味着编译器可以交换这两个操作数，如果这是使所有操作数符合约束条件的最便宜的方法。`%` 适用于所有的替代方案，并且必须作为约束条件中的第一个字符出现。只有只读的操作数可以使用 `%`。

  GCC 在一个 `asm` 中只能处理一个交换对；如果使用更多，编译器可能会失败。需要注意的是，如果两个选项严格意义上是相同的，就不需要使用修饰符；这只会在重载过程中浪费时间。
