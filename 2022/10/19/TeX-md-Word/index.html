TeX, Word & Markdown

2022-10-19
LaTeX-LaTeX排版

TeX、Word 和 Markdown，本质上没法并列到一起去。TeX 是排版软件，Word 是字处理软件，Markdown 充其量是 HTML 里 <h1> <h2> 这类标签的抽象。但总之它们三个都能写文档，甚至有的期刊同时接受 (La) TeX、Word 和 Markdown 投稿，并列到一起也不无道理。

这里的 Word 指 Microsoft Word 及其同类产品（如 LibreOffice Write），但不包括国产败类 WPS Office（WPS 你 妈什么时候死啊）。

内容与样式

TeX 最为称道的一点是内容与样式分离，然而这通常是 LaTeX 或者 ConTeXt 的事。在 plain TeX 上，你还是得操心字号、字重、编号什么的。要去操心寄存器分配 —— 这一点不比汇编的难度小。但总之今天也没有人，特别是中国人，用 plain TeX 写正经文档了（不加载 xeCJK 还想要标点禁则？pTeX 佬除外）。

LaTeX 的内容与样式分离已经相当好了。你可以设置一个章节标题格式，然后每次 \chapter{} 让它自动增加计数器、画出章节标题、开始新的一章就好。也可以自行设定页眉页脚，甚至奇偶页不同。当然，大佬们也可以用一些底层的 TeX 命令干预排版，或者 Lua (Meta) TeX 那种用 Lua 回调控制排版。总之 (La) TeX 是自由的。

Markdown 是完全的内容与样式分离，内容上只是 Markdown 语法到标准 HTML 标签的转换，样式完全由 CSS 定义。你正在读的这篇文章，用 Hexo 的 yun 主题渲染，和用 Typora 的嘉然主题渲染，完全是两个观感。

但在 Word 里想要打出标题，能选中文本然后选择 “标题 1” 的时候，也可以选中文本→粗体→调字号→居中，不会设置居中的甚至可以用空格挤到居中… 然后再碰上秉承 “能看就行” 的用户，就得到了一份令人血压升高的文档。

这就说到下面的问题：上手难度。

上手难度

Word 是没有上手难度的。打开 Word，只要会打字，总能做出来点什么来。不会配置样式，就手动调字号调字体。想要某个功能但不知道在哪里，胡乱遍历一遍菜单总能找到。不会设置居中？用空格挤过去。不会断页？用一堆回车挤过去。

Markdown 的上手难度也不过是语法，但就那么一丁点玩意… 背不下来语法说不过去吧？如果配合上萌新友好的编辑器（如 Typora，尽管对钱包不友好），甚至不会任何 md 语法都能写出 md 文档。

当然，不会 md 语法，把 md 编辑器当纯文本编辑器，也总能输出点什么。

即便不会大部分语法，plain TeX 也总能输出点什么，但文档末尾要有 \bye——plain TeX 只为有礼貌的人工作。

LaTeX 是不一样的。一群猴子敲打字机或许能弄出来 \documentclass{article} \begin{document} Monkey! \end{document}，但一个刚接触 LaTeX 的萌新肯定不会。

所以，对于纯新手，我们有不等式：Word > md > LaTeX.

但是，一旦学会了 md 语法，结合预设的漂亮的 CSS，就能写出漂亮的文档。一旦学会了 \title{}、\section{} 一类的命令，结合优雅的标准文档类，就能输出漂亮的文档：用上 ctexart，再用上 gb7714-2015，保证你的文章有正经论文的味。但 Word 特有的易上手以及过于自由的样式控制，使得新手很容易胡乱去调字体。对于有一定基础的人，我们有不等式：LaTeX > md > Word。

对于真正的大佬，在 LaTeX 里能用核心命令和引擎原语干预排版，在 md 里能手搓 HTML 标签，在 Word 里熟练运用 VBA 和域代码，那用什么也就无所谓了。

但这样的大佬很少。大多数人都不过是有一定基础，可能在三者中的某一个精进。于是：

  • 给 md 写主题，需要熟知 CSS ，与 HTML，偶尔甚至研究 JS。这远超大多数人的能力。
  • 给 (La) TeX 写宏包写文档类，要熟知 (La) TeX 编程，熟知引擎原语，熟知 PDF，OpenType 等的特性，这远超大多数人的能力。
  • 给 Word 弄模板… 只需要有一点基础。

然后就是萌新制作 Word 模板的群魔乱舞时间（某世界一流大学 AI 引论课程的模板）：字体群魔乱舞，正文的英文部分全是等宽宋体，标题编号用 Calibri，参考文献用 Times New Roman，标题又用上了微软雅黑。

正文字号也是五号、小四、小三、15.5 pt 群魔乱舞，一不小心就跳到另一个字号。还有极为精妙的左侧 0.19 cm，右侧 0.44 cm，首行 0.74 cm 缩进配段前 14.75 pt 间距，它的下一个段又是左侧 0.38 cm，右侧 0.31 cm，首行 0.56 cm 缩进，段前 0.3 pt 间距，再下一段又变成左侧 0.19 cm，右侧 0.31 cm，首行 0.56 cm 缩进… 这个模板还通过玄学关掉了标点压缩和字距调整，实在优雅！实在精妙！我只觉自己太差太差，看不透这高深的模板。

所以，我们既希望软件能零难度上手，又觉得用一定的上手难度挡住傻逼还是有必要的。起码 LaTeX 不会有人做出这样的睿智模板还发给大家用，TeX 引擎会先吐槽说 badness 10000。

一定的上手难度可以逼迫人去看新手教程，比如 lshort-zh-cn。
中学的信息技术课，大抵不会教怎么使用 Word 的样式罢。

“图灵完备”

某种意义上说，Word 的 VBA 功能更强大，但代码即数据的 TeX 宏语言（有点 Lisp 的味道？）或许更适合文档处理。例如，对于补集的符号 ~（某奇妙教材的奇妙符号），我希望当后面是字母的时候，在中间插一个 -1/6 em 的空格，当后面是括号的时候什么都不做。LaTeX 就可以简单地做到这一点：

\makeatletter
\renewcommand{\complement} {
    \mathop{\sim}
    \@ifnextchar({}{\!}
}
\makeatother

Word 显然是无能为力的。LaTeX 里想要一个适用各种情景的微分算子？看看 fixdif 包的实现就知道了。Word 是断不能实现的。

VBA 脱胎于正统编程语言 VB，而 TeX 这边也有 LuaTeX 和 LuaMetaTeX。实例是 TikZ 的 Lua 数学计算后端。

至于 md，是没有图灵完备一说的。但 md 简单的语法，倒也便于用 Shell 或者 Python 脚本去自动化生成。

字体

Markdown 这边只能拿 CSS 调字体。Word 倒是可以方便地用 GUI 调字体，但如果你试过把思源宋体从 Extra Light 到 Heavy 全字重都装上，就会发现有几个字重不在列表里面。

Word 对于粗体和斜体是优先寻找对应的变体，没有的情况下会用伪粗体或伪斜体。Markdown 也是一样的，但毕竟是解析到 HTML，现代浏览器对可变字体的支持还是很好的（如果你在 CSS 里用了 Weight 轴可变的字体）。

MS Word 对于三次曲线的字体的支持不甚良好。导出到 PDF 的时候，会转换成位图。LibreOffice Write 在这方面就很好。

TeX 默认的西文字体是 Computer Modern。LaTeX 的中文字体主要看 ctex 包，一般来说，Windows 下是中易宋体。

TeX 发行版往往打包了相当多的字体，TeX Live 那几个 GB 里，字体是大头。事实上，直到本世纪，才有了能直接调用系统字体的 XeTeX，LuaTeX 随后跟进。

然而调用系统字体在一些情况下拖慢了编译速度，比如要刷新字体缓存的时候。

一句话总结：不同的 TeX 引擎，对字体调用的处理是不同的，没有放之四海而皆准的方案。

中易系的字体，例如说（新）宋体、黑体、楷体，它们的西文部分完全是凑数的 —— 或许是为了强行等宽？Visual Studio 的中文版，默认字体还是宋体。但碰上没有经验和审美的用户，就糟了：全选→字体→宋体，然后就看见西文部分也变成了等宽的宋体：

  • 中易宋体 / 新宋体 / 仿宋的西文部分。这三者西文部分完全相同，仿宋的基线稍有抬高。
  • 中易黑体的西文部分。
  • 中易楷体的西文部分。
  • 思源宋体的西文部分，事实上取自 Source Serif Pro。Regular 字重。
  • 思源黑体的西文部分，事实上取自 Source Sans Pro。Regular 字重。

还是那句话，中易系的字体，西文部分就是个凑数的。好在国内的字库公司逐渐注意到了中西文字体搭配的问题。

然后说说方正。用方正系软件排版的数学书简直是灾难。可以自行对比《数学分析》（伍胜建，TeX 排版）和《离散数学教程》（耿素云等，方正系软件排版）。

数学排版

TeX 对数学排版是有专门设计的，TeX 的 16 个 catcode（有的引擎扩展到 20 个）里，3 表示 math shift，7 表示 superscript 而 8 表示 subscript。再搭配上专门设计的 Computer Modern Math 字体，TeX 能排版优美的数学公式。

Word 对数学公式也有专门的设计，早期是域代码和内嵌 MathType，现在是专门的 OfficeMath。这主要依赖 OpenType 的 Math 表，而 Math 表又是受 TeX 启发，其目标之一便是实现 TeX 级别的公式排版。

Markdown 通常使用 MathJax 或者 KaTeX 这类 JS 库实现数学公式排版，这与编辑器有关。比如 MarkText 的行内公式甚至连中线都对不齐（恼）。

人们喜欢 TeX 公式的一个原因是可以使用标准键盘输入符号，比如可以输入 \sum 而不是要在输入法里找∑。但很有意思的是，现在 LaTeX 可以用 unicode-math 包来直接输入∑∫这类符号，Word 也允许输入 \sum、\int 并配合自动更正来输入符号。

我还想说说 LibreOffice Math，这玩意倒是自己弄了一套标记语言，可以用键盘输入符号，GUI 也做得很不错，特别是还能嵌入 Write、Calc、Impress 这些程序。

中文排版

Word 是早就支持中文排版了。割注、直排这类东西也很早就支持了。不过我倒是很疑惑，PowerPoint 和 Word 用的到底是不是同一套排版引擎？PowerPoint 的字距调整跟没有一样，LibreOffice Impress 的字距调整就很正常。

TeX 这边，首先是 CCT，TY 等引擎，然后是 CJK 宏包，再接下来是 XeLaTeX 和 xeCJK 包，XeTeX 解决了编码问题和系统字体调用问题，xeCJK 用宏解决了标点处理和字距调整，以及中英文字体分别调用。缺点是一堆宏实在拖慢编译速度… LuaLaTeX 上，是基于 LuaTeX-ja 包弄的中文支持。而 pTeX 系则是原生支持 CJK。这些解决方案最后被 ctex 宏集封装。

直排这个东西，大陆是没什么需求的，除非是搞古文。但是港台和日本是有相当需求的。上世纪就有了 pTeX 这个东西，然后是其 Unicode 版 upTeX。pTeX 系引擎引擎在底层就支持了标点处理、字距处理和直排（Pascal/C 编译的机器码可比 xeCJK 的宏展开快多了）。最新的 LuaTeX-ja 也开始支持直排了。XeTeX 搞直排只能靠邪招，比如页面逆时针旋转 90°，然后文字顺时针旋转 90° 这样子。

兼容性？

Word 的兼容性是众所周知的… 好吧我也不知道怎么说。一方面，现在的 Word 还可以打开上古的.doc 文档，另一方面，不同版本的 Word 打开同一份文件，在排版上也可能有区别，在不同的电脑上，由于系统、软件、字体的不同，更可能出问题（WPS 你 妈什么时候死啊）。辛辛苦苦做的文档在老板的电脑上一片空白。

导出成 PDF 不就没事了嘛

TeX 这边，Knuth TeX 的设计早已冻结，然而其他引擎和宏包的开发还是热火朝天的。LaTeX 3 的最终细节还没定下来，试图依赖不稳定 API 的宏包可能会遭殃。另一方面，不同引擎的不同原语，也会造成兼容性问题。比如你不可能在 XeTeX 上用 pdfTeX 在 PDF 输出上的原语，也不可能在 pTeX 上用 LuaTeX 的 \directlua。

但很多宏包会对不同引擎做适配，或者直接说明依赖特定引擎，比如 fontspec 要求必须是 XeLaTeX 或 LuaLaTeX。LaTeX 很大程度上也抹平了不同引擎的差异。

Markdown 的兼容问题在于各大编辑器 / 渲染器的语法不统一，渲染流程不统一。比如 Typora 用 MathJax 渲染数学公式，而 MarkText 用 KaTeX 渲染… 这两个引擎的语法不统一，从一个换到另一个，你的文档很可能就炸了。

而且标准 Markdown 没有数学公式的语法，都是各家扩展的。还有删除线、高亮一类语法。还有渲染器特定的语法，比如 Hexo 的标签，放到别的地方直接爆炸。又如我上学期的小组作业 HomoDark，直接移除了表格和数学公式支持。

三者的伟大之处

TeX 最伟大的地方在于，把铅活字时代排字工的技艺以 Knuth-Plass 算法自动化，固化并传承了下来。其他很多软件都丢失了这些技艺。不是说 “告别铅与火，走向光与电” 不好，但是别把以前的好传统丢了啊… 为什么 Knuth 要开发 TeX？就因为早期的计算机排版实在太差，根本比不上排字工。

Word 的伟大之处在于让零基础的人也能做出文档，而且是所见即所得的。早期的 Word 比求伯君的 WPS 好了十分甚至九分。

Markdown 的伟大之处在于为结构化层次化的文档提供了极简的语法，即便没有渲染器也可以轻松阅读。特别适合理工科，尤其是计算机领域文档的编写。而且 md 是完全的内容与样式分离。

TeX 的未来？

有人觉得 TeX 社区是一群守旧的人。但真正的 TeX 用户比其他人更希望革了 TeX 的命，TeX 的替代品，社区已经等了几十年了。从 NTS、Omega、Aleph，到 LuaTeX、LuaMetaTeX，以及 pTeX 系引擎为东亚文字排版做出的努力，这都是 TeX 的进步。很多人知道 TeX 是 1982 年的产品，但很多人不知道的是 Knuth TeX 早就进了博物馆，大家在用的其实是 NTS 项目的遗留：e-TeX。

或许十年内就能看见内嵌 Python 解释器的 TeX 引擎吧，或许内嵌 JS 的引擎也快出现了吧…TeX 宏语言的本质应该是不会动的，习惯宏语言的就写 TeX，不习惯宏语言的就靠内嵌脚本吧…
